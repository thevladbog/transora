# Transora. Стартовая архитектура и технологический стек

Дата: 2026-06-13  
Статус: черновое архитектурное решение для старта разработки

## Контекст

Transora - система управления сетью автовокзалов: расписания, рейсы, места, продажи, возвраты, посадка, документы, табло, объявления и интеграции с локальным оборудованием.

Документы в `docs/` описывают хорошую доменную декомпозицию, но предложенная там микросервисная архитектура слишком тяжелая для старта. Для MVP важнее быстрее получить рабочую систему с понятными границами модулей, надежными транзакциями и возможностью позже вынести части в отдельные сервисы без переписывания доменной модели.

## Решение

Стартуем с Kotlin-first архитектуры:

- Core backend: модульный монолит на Kotlin + Spring Boot.
- Реальные отдельные процессы с первого дня: `station-agent`, `hardware-agent`, мобильное приложение посадки.
- Хранилище MVP: PostgreSQL как основной источник истины.
- Асинхронность внутри core: transactional outbox в PostgreSQL.
- Внешние брокеры, API Gateway и Kubernetes закладываем архитектурно, но не делаем обязательными для первого рабочего релиза.

Модульный монолит означает, что bounded contexts остаются отдельными модулями кода и схемами ответственности, но деплоятся одним backend-приложением.

## Актуальный базовый стек

Используем актуальные стабильные версии на дату фиксации решения. Не используем плавающие теги `latest` в Docker, Gradle plugins, base images и dependency declarations.

| Компонент | Версия / решение | Назначение |
| --- | --- | --- |
| Java | 25 LTS | runtime для backend |
| Kotlin | 2.4.0 | основной язык backend и Android |
| Spring Boot | 4.1.0 | backend framework |
| Gradle | 9.5.1 | сборка и управление проектом |
| PostgreSQL | 18.4 | основная БД |
| Flyway | актуальная стабильная | миграции БД |
| Spring Security | версия из Spring Boot BOM | безопасность backend |
| OpenAPI | springdoc-openapi или совместимый актуальный стек | контракт REST API |
| Docker Compose | актуальная стабильная | локальная разработка и MVP deployment |

Источники для проверки версий:

- https://spring.io/projects/spring-boot
- https://kotlinlang.org/docs/releases.html
- https://gradle.org/releases/
- https://www.postgresql.org/support/versioning/
- https://www.oracle.com/java/technologies/java-se-support-roadmap.html

Правила обновления:

- версии фиксируются явно;
- обновления проходят через Renovate или Dependabot;
- beta, RC, EAP и snapshot версии не используются без отдельного решения;
- для критичных библиотек берем версии из Spring Boot BOM, если это совместимо с задачей;
- перед переходом на новую major-версию делаем отдельный ADR.

## Границы core-модулей

Внутри backend выделяем модули:

- `iam`: пользователи, роли, права, сессии, токены.
- `scheduling`: маршруты, остановки, расписания, рейсы, перевозчики, автобусы, водители.
- `inventory`: схема мест, доступность, бронирования, квоты, ограничения.
- `sales`: смены, заказы, билеты, оплаты, возвраты, тарифы.
- `documents`: билеты, посадочные ведомости, отчеты, печатные формы.
- `boarding`: проверка билетов и события посадки.
- `notifications`: события для табло, объявлений и клиентских каналов.
- `admin`: настройки станций, справочники, операционные панели.

Модули не должны превращаться в "папки без границ". Каждый модуль владеет своими use cases, доменными правилами и таблицами. Другие модули обращаются к нему через публичные application-интерфейсы или события, а не через прямое изменение чужих таблиц.

## Физические границы

С первого дня отдельно от core живут:

- `station-agent`: локальный агент автовокзала, кэш расписаний, режимы degraded/offline, связь с табло и локальными устройствами.
- `hardware-agent`: локальный агент рабочего места кассира, доступ к ККТ, платежному терминалу и принтеру билетов. Доступен только с localhost или из доверенного локального контура.
- `boarding-app`: Android-приложение для инспектора посадки.

Это не преждевременная микросервисность: эти компоненты отделяются не ради масштаба, а потому что находятся в другой сети, зависят от оборудования или должны работать при нестабильной связи.

## Интеграционный стиль MVP

Основной внешний API - REST + OpenAPI.

Для внутренних событий core используем transactional outbox:

1. Доменная операция меняет состояние в PostgreSQL.
2. В той же транзакции пишется запись в outbox.
3. Отдельный publisher читает outbox и доставляет событие подписчикам.

На MVP события могут обрабатываться внутри того же приложения. Позже outbox publisher можно направить в NATS без изменения доменных операций.

Не публикуем события напрямую из бизнес-транзакции в брокер. Сначала фиксируем состояние в БД, затем доставляем событие.

## Инварианты, которые нельзя ломать

- Продажа невозможна без активной резервации мест.
- Продажа невозможна без открытой кассовой смены.
- `sales` не меняет статус места напрямую; он работает через `inventory`.
- Резервация места должна быть защищена от гонок.
- Возврат денег и фискализация не откатываются простым rollback после успешного внешнего действия.
- Нельзя отменить рейс, если есть активные билеты, без явной процедуры возврата или переноса.
- `hardware-agent` не публикуется наружу.
- Секреты не хранятся в репозитории.
- Оффлайн-продажи на старте запрещены; оффлайн допускается для просмотра кэша и посадочных сценариев с последующей синхронизацией.

## Данные

PostgreSQL - основной источник истины. На старте предпочтительнее одна база с разделением по схемам или четкими префиксами таблиц модулей. Это упрощает транзакции в MVP, но сохраняет границы владения данными.

Redis не является обязательным базовым компонентом MVP. Его добавляем, когда появится подтвержденная потребность:

- распределенные блокировки на горячих рейсах;
- краткоживущие токены и blacklist;
- кеши табло или публичного расписания;
- очереди локальных уведомлений.

MinIO или S3-совместимое хранилище добавляем, когда появятся PDF-документы, отчеты или медиа для объявлений.

## Deployment

Для разработки и первого внедрения:

- Docker Compose;
- один backend;
- PostgreSQL;
- web frontend;
- локальные агенты на станциях и рабочих местах.

Целевое развитие:

- Kubernetes;
- отдельный API Gateway;
- NATS для межсервисных событий;
- горизонтальное масштабирование публичных и кассовых API;
- HA PostgreSQL;
- централизованные логи, метрики и трассировка.

Переход к микросервисам рассматриваем только после появления реальной причины: независимый масштаб, независимый цикл релизов, разные SLA, отдельная команда владения или интеграционная изоляция.

## UI направление

Figma-макет `Bottle Code v.2` хорошо подходит как визуальная база для табло, киосков и публичных экранов: темная тема, крупная типографика, сильный синий акцент, ясные статусы рейсов.

Для диспетчеров, кассиров и администраторов используем более плотный операционный интерфейс: таблицы, формы, быстрые фильтры, статусы, минимум декоративности. Визуальная связь с Figma сохраняется через цвета, типографику и компоненты, но рабочие панели не должны выглядеть как публичное табло.

## Ближайшие следующие решения

1. Зафиксировать структуру репозитория: один Gradle multi-module проект или несколько приложений рядом.
2. Выбрать frontend stack для web: React/Vite, Next.js или другой вариант.
3. Описать доменную модель MVP: станции, маршруты, рейсы, автобусы, места, смены, заказы, билеты.
4. Спроектировать первый vertical slice: создание рейса, резервирование места, продажа билета, печать или генерация документа.
5. Подготовить локальный `docker-compose.yml` для PostgreSQL и backend.
