## 1. Общее описание системы

Система управления автовокзалами — это централизованная платформа для автоматизации операционной деятельности сети автовокзалов. Система построена по принципу **«ядро + подключённые вокзалы»**: существует центральный узел (ядро), к которому подключаются отдельные автовокзалы и работают в рамках своего операционного контура.

### Основные задачи системы

- Централизованное управление расписаниями движения с синхронизацией изменений по всем вокзалам в реальном времени
- Продажа билетов на физических кассах вокзала с поддержкой схем мест автобусов
- Контроль инвентаря мест с системой блокировок и квот по вокзалам
- Автоматическое формирование поездной документации (манифесты, посадочные ведомости)
- Учёт посадки пассажиров через мобильное приложение для терминалов сбора данных (ТСД)
- Управление табло прибытия/отправления и перронными табло
- Автоматизированная система озвучивания сообщений (рейсовые, информационные, рекламные)

### Масштаб

Система проектируется для начального развёртывания на **до 10 вокзалов** с возможностью горизонтального масштабирования без изменения архитектуры.

## 2. Ключевые принципы архитектуры

**Единый источник правды.** Ядро системы является единственным авторитетным источником данных о рейсах, расписаниях и инвентаре мест. Все вокзалы работают с данными ядра.

**Событийная синхронизация.** Любое значимое изменение (опоздание рейса, открытие продаж, блокировка мест) публикуется как событие. Все заинтересованные компоненты системы — вокзалы, табло, система озвучивания — получают обновление автоматически.

**Изоляция операционных контуров.** Каждый вокзал работает в своём контуре. Диспетчер видит и управляет только теми рейсами, которые обслуживает его вокзал.

**API-first.** Все сервисы взаимодействуют через задокументированные API. Это обеспечивает возможность подключения онлайн-продаж, мобильных приложений и сторонних интеграций в будущем.

**Аудит всех действий.** Каждое изменение данных — блокировка места, ручная правка рейса, открытие продаж — фиксируется в неизменяемом журнале аудита с указанием оператора, времени и вокзала.

## 3. Участники системы и роли

### Администратор системы
- Управление вокзалами, пользователями и ролями
- Настройка тарифов и правил возвратов
- Доступ к сводной отчётности по всей сети

### Диспетчер вокзала
- Управление расписанием в рамках своего вокзала
- Фиксация фактических времён прибытия/отправления
- Управление блокировками продаж на вокзале
- Обслуживание проходящих рейсов (внесение данных по прибытию, открытие/закрытие продаж)
- Управление очередью сообщений системы озвучивания

### Кассир
- Продажа и возврат билетов
- Работа со сменой ККТ
- Просмотр расписания рейсов

### Водитель / Агент автопарка
- Предоставление информации о составе рейса (ТС, схема мест, занятые места) при обслуживании проходящих рейсов

### Перевозчик (Автобусный парк)
- Предоставление информации об автобусах, схемах мест, водителях
- Просмотр отчётов по своим рейсам

## 4. Функциональные модули

### 4.1 Планирование рейсов

Модуль обеспечивает полный жизненный цикл рейса: от создания маршрута и расписания до фиксации фактических данных движения.

**Маршруты**
- Создание и редактирование маршрутов с указанием списка остановок, порядка следования и нормативного времени между остановками
- Поддержка остановок вне системы (для проходящих рейсов)
- Привязка маршрута к перевозчику

**Расписание**
- Три типа расписаний: постоянное, сезонное (с датами начала и окончания), исключение (на конкретную дату)
- Создание шаблонов рейсов с привязкой к расписанию
- Назначение ТС и водителя на рейс (вручную или через импорт данных от автопарка)

**Рейс (конкретная дата)**
- Статусы рейса: `Запланирован → Активен → В пути → Прибыл → Завершён / Отменён`
- Фиксация фактических времён прибытия и отправления по каждой остановке
- Фиксация опозданий с автоматическим уведомлением всех вокзалов рейса
- Замена ТС или водителя по факту

**Транспортные средства**
- Хранение данных об автобусах: модель, госномер, схема расположения мест
- Привязка схемы мест к конкретному ТС
- Данные поступают от автопарка и редактируются диспетчером при необходимости

**Водители**
- Хранение ФИО, номера водительского удостоверения
- Назначение на рейс

### 4.2 Управление инвентарём мест

Это центральный и наиболее критичный сервис системы с точки зрения консистентности данных.

**Инвентарь для каждого рейса**
- Схема мест автобуса с визуальным отображением
- Статус каждого места: `Свободно / Зарезервировано (TTL) / Продано / Заблокировано`
- Резервирование места при начале оформления с таймером истечения (10 минут)
- Автоматическое освобождение при истечении резерва без оплаты

**Схема автобуса**
- Визуальное отображение раскладки мест в кассовом интерфейсе
- Отображение статуса мест в реальном времени
- Поддержка различных типов мест (обычное, верхняя/нижняя полка и т.д.)

### 4.3 Продажа билетов

**Кассовый интерфейс**
- Поиск рейсов по направлению и дате
- Визуальный выбор места на схеме автобуса
- Ввод данных пассажира (ФИО, тип и номер документа)
- Расчёт стоимости с учётом тарифов
- Выбор способа оплаты: наличные или банковская карта
- Печать билета на термопринтере
- Фискализация через подключённую ККТ

**Продажа нескольких билетов**
- Возможность оформить несколько билетов в рамках одной транзакции (группа пассажиров)

**Контроль доступа к продажам**
- Кассир может продавать только те билеты, которые доступны на его вокзале согласно правилам блокировок

### 4.4 Система возвратов

**Расчёт удержаний**
Штрафы за возврат рассчитываются согласно Постановлению Правительства РФ №112 и настраиваемым правилам:

| Срок до отправления | Удержание |
|---|---|
| Более 24 часов | 0% |
| От 2 до 24 часов | 10% |
| Менее 2 часов | 25% |
| После отправления | Возврат не производится |

- Параметры удержаний настраиваются администратором системы
- При возврате автоматически рассчитывается сумма к выдаче
- Место освобождается в инвентаре сразу после подтверждения возврата
- Фискальный возвратный чек формируется через ККТ

### 4.5 Формирование поездной документации

Документы формируются автоматически по событию начала посадки на рейс.

**Манифест рейса**
- Полный список пассажиров с данными документов
- Информация о рейсе, маршруте, ТС и водителе
- Формат: PDF для печати и/или электронный

**Посадочная ведомость**
- Список пассажиров с местами
- Штрихкод или QR-код каждого билета для сканирования при посадке
- Формат для печати (передаётся водителю)

**Отчёты для перевозчика**
- Отчёт по рейсу: количество пассажиров, выручка по рейсу
- Формируется по завершении рейса

### 4.6 Учёт посадки пассажиров

**Мобильное приложение для ТСД (Терминал сбора данных)**
- Работает на Android-устройствах (промышленные ТСД или смартфоны)
- Сканирование штрихкода / QR-кода на билете
- Мгновенная проверка: билет действителен / уже использован / не существует
- Отображение данных пассажира и номера места
- Офлайн-режим: кеш списка пассажиров рейса с последующей синхронизацией
- Подсчёт посаженных пассажиров в реальном времени

### 4.7 Табло рейсов и перронов

**Табло прибытия/отправления (главное)**
- Веб-приложение, выводимое в полноэкранном режиме на телевизор или монитор
- Отображает актуальный список рейсов с фактическими временами
- Обновление в реальном времени (WebSocket)
- Индикация статуса: по расписанию / опаздывает / отменён / посадка / отправился

**Перронное табло**
- Аналогичное веб-приложение, настроенное на конкретный перрон
- Отображает информацию о текущем рейсе на перроне
- В перспективе: поддержка диодных табло через специализированный протокол

**Конфигурация табло**
- Каждое табло настраивается на конкретный вокзал и (для перронных) перрон
- Настройка фильтров отображения: только отправления, только прибытия, оба

### 4.8 Система озвучивания

**Принцип работы**
Система формирует непрерывный поток аудиосообщений, распределяя их по приоритетам:

1. **Рейсовые сообщения** (высший приоритет) — автоматически генерируются на основе событий расписания
2. **Информационные сообщения** — вносятся диспетчером вручную
3. **Рекламные сообщения** — воспроизводятся в фоновом режиме при отсутствии сообщений более высокого приоритета

**Автоматические рейсовые триггеры**
- Рейс отправляется через 30 минут → объявление о начале регистрации
- Начало посадки → объявление о посадке с указанием перрона
- Задержка рейса → объявление о задержке и новом времени
- Отмена рейса → объявление об отмене

**Синтез речи**
- При наличии записанного аудиофайла для шаблона → воспроизводится запись
- При отсутствии записи → текст передаётся в TTS (Yandex SpeechKit)
- Синтезированный аудиофайл сохраняется в кеш для повторного использования
- Fallback при недоступности Yandex SpeechKit → локальный TTS (RHVoice)

**Управление очередью диспетчером**
- Просмотр актуальной очереди сообщений
- Вставка внеочередного сообщения (с немедленным воспроизведением или после текущего)
- Отложить или пропустить запланированное сообщение
- Ввод произвольного текста для разового озвучивания

### 4.9 Управление оборудованием кассы

Кассовое место взаимодействует с физическим оборудованием через локальный Hardware Agent.

**Поддерживаемое оборудование**
- **ККТ (онлайн-касса):** АТОЛ, Штрих-М и совместимые. Открытие/закрытие смены, печать фискального чека, Z-отчёт
- **Банковский терминал:** Ingenico, VeriFone, PAX. Проведение оплаты картой, возврат по транзакции
- **Принтер билетов:** Термопринтер (ESC/POS, ZPL). Печать билета, проверка наличия бумаги
- **Сканер штрихкодов:** Для считывания билетов при возврате или обмене

**Атомарность операций**
При оплате картой и фискализации применяется механизм компенсирующих транзакций (Saga): если один из шагов завершился ошибкой, система инициирует отмену предыдущих шагов с уведомлением кассира.

## 5. Модель расписаний

Расписание является общим ресурсом системы — единым для всей сети вокзалов.

**Иерархия данных расписания**

```
Маршрут (Route)
  ├── список остановок с нормативным временем
  └── принадлежит перевозчику

Шаблон расписания (Schedule)
  ├── тип: ПОСТОЯННОЕ | СЕЗОННОЕ | ИСКЛЮЧЕНИЕ
  ├── период действия (dateFrom, dateTo)
  ├── дни недели или конкретные даты
  └── набор рейсов по шаблону

Рейс (Trip) — конкретная дата
  ├── привязка к шаблону расписания
  ├── назначенное ТС и водитель
  ├── нормативные времена по каждой остановке
  ├── фактические времена (заполняются по ходу)
  └── статус
```

**Синхронизация изменений**

Любое изменение рейса (опоздание, отмена, смена ТС) публикуется как событие в шину сообщений. Все вокзалы, обслуживающие данный рейс, получают обновление немедленно и отражают его на табло, в очереди объявлений и в интерфейсе диспетчера.

**Пример:** Диспетчер вокзала A фиксирует опоздание рейса Москва–Красноярск на 40 минут. Вокзалы B, C и D, через которые проходит этот рейс, автоматически получают обновление и отображают новое расчётное время прибытия на своих табло.

## 6. Система блокировок продаж

Система блокировок регулирует, какие места на каком вокзале могут быть проданы.

### 6.1 Квоты по вокзалам

Для каждого вокзала и рейса можно настроить диапазон доступных для продажи мест.

**Пример:** На вокзале «Красноярск» для рейса «Москва–Красноярск» разрешена продажа только мест с 40 по 45.

**Параметры квоты:**
- Вокзал
- Рейс или шаблон расписания (для повторяющихся рейсов)
- Диапазон разрешённых мест (или список конкретных мест)
- Статус: активна / приостановлена

**Управление:** Диспетчер вокзала может в любой момент заблокировать или разблокировать квоту. Изменение применяется немедленно. Для повторяющихся рейсов изменение применяется ко всем будущим рейсам шаблона, либо только к конкретному рейсу.

### 6.2 Блокировка после завершения посадки

После завершения посадки на одном вокзале диспетчер может заблокировать продажу на всех последующих остановках маршрута. Это предотвращает продажу мест, которые физически уже заняты.

## 7. Обслуживание проходящих автобусов

Проходящий автобус — рейс, который следует через вокзал транзитом (не начинается и не заканчивается на данном вокзале), и до момента прибытия реальная вместимость (занятые места) неизвестна.

**Процесс обслуживания:**

1. **До прибытия:** Рейс отображается в расписании. Продажи заблокированы (статус: «Ожидается прибытие»).
2. **Прибытие автобуса:** Диспетчер фиксирует факт прибытия в системе.
3. **Внесение данных:** Диспетчер вносит список свободных мест по информации от водителя. Схема мест автобуса отображается визуально.
4. **Открытие продаж:** Диспетчер открывает продажу на указанные свободные места. Кассиры немедленно получают доступ к продажам.
5. **Завершение посадки:** Диспетчер закрывает посадку. Продажи на последующих вокзалах блокируются (опционально).

## 8. Работа при потере связи

При потере соединения вокзала с ядром системы применяются следующие правила:

| Функция | Поведение при потере связи |
|---|---|
| Продажа билетов | **Полная остановка.** Касса отображает статус «Нет связи». |
| Возврат билетов | **Остановка.** |
| Табло рейсов | **Продолжает работу** на основе последнего полученного кеша расписания |
| Система озвучивания | **Продолжает работу** по сохранённому расписанию и кешированным аудиофайлам |
| Учёт посадки (ТСД) | **Офлайн-режим:** работает по кешу, синхронизация после восстановления связи |
| Рабочее место диспетчера | Только просмотр кешированных данных, изменения недоступны |

Остановка продаж при потере связи обусловлена требованиями российского законодательства в части фискализации и контроля инвентаря мест.

## 9. Интеграции

### Текущие (на старте)

| Система | Тип интеграции |
|---|---|
| ОФД (Оператор фискальных данных) | Передача фискальных данных через ККТ. Платформа ОФД / СБИС |
| ККТ (АТОЛ, Штрих-М) | Прямая интеграция через SDK производителя |
| Банковские терминалы (Ingenico, VeriFone, PAX) | SDK или OPOS |
| Yandex SpeechKit | REST API для синтеза речи |

### Планируемые (следующие итерации)

| Система | Описание |
|---|---|
| Онлайн-продажи (сайт/приложение) | API для внешних каналов продаж |
| Агрегаторы билетов | Интеграция инвентаря и продаж |
| Система мониторинга транспорта (GPS/AVL) | Автоматическое обновление статуса рейса по местоположению |
| Диодные табло | Управление через специализированный протокол |

## 10. Технологический стек

### Бэкенд

| Компонент | Технология | Обоснование |
|---|---|---|
| Бизнес-сервисы ядра | **Kotlin + Spring Boot** | Надёжная экосистема, ACID-транзакции, JasperReports |
| Системные сервисы | **Go** | Лёгкий деплой, высокая производительность для I/O |
| Station Agent (вокзал) | **Go** | Один бинарник, управление оборудованием, минимальные зависимости |
| Hardware Agent (касса) | **Go** | Прямая работа с COM/USB устройствами |

### Инфраструктура

| Компонент | Технология | Обоснование |
|---|---|---|
| API Gateway | **Kong** | Маршрутизация, аутентификация, rate-limiting |
| Event Bus | **NATS JetStream** | Лёгковесный, надёжный, встроенный persistence, один бинарник |
| База данных | **PostgreSQL** | ACID, надёжность для транзакционных данных |
| Кеш / Сессии | **Redis** | TTL-блокировки мест, сессии касс |
| Деплой ядра | **Kubernetes** | Масштабируемость, высокая доступность |
| Деплой вокзала | **Docker Compose + systemd** | Простота эксплуатации на точке |
| Мониторинг | **Grafana + Prometheus + Loki** | Зрелый open-source стек |

### Фронтенд

| Компонент | Технология |
|---|---|
| Кассовое место | **React + TypeScript + Tauri** |
| Рабочее место диспетчера | **React + TypeScript** (веб) |
| Табло рейсов | **React** (fullscreen веб-приложение) |
| Перронное табло | **React** (fullscreen веб-приложение) |
| Приложение для ТСД | **Android Native** (Kotlin) |

### Внешние сервисы

| Компонент | Технология |
|---|---|
| TTS (основной) | **Yandex SpeechKit** |
| TTS (fallback) | **RHVoice** (локальный, офлайн) |
